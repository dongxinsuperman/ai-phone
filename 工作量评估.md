# ai-phone 工作量评估（手写实现视角）

> 本文假设由**一名 3-5 年经验的测试开发工程师**（非纯后端/前端专精）手动实现本项目所有功能（v1.0 → v1.5），不借助 AI 代码生成。
>
> **估时口径**：
> - 单位：人天 = 8 小时
> - **包含**：方案调研、踩坑、真机联调、关键 bugfix
> - **不包含**：正式测试团队的系统测试
> - 小于等于 1 天的事项 → 算 1 项
> - 大于 1 天且不可再拆的功能 → `N/M` 表示（如 `1/3` = 该功能共 3 天，这是第 1 天）
> - **VLM 核心决策循环** 视为从其他项目迁移过来的资产，只算适配成本
>
> **工程师能力画像**：
> - 熟悉 adb / scrcpy / 基础 Python API 开发
> - 陌生栈：FastAPI 深度 / WebSocket 并发 / FFmpeg + MSE / VLM prompt 工程 / Xcode 签名 / HarmonyOS 协议
> - 前端是弱项：能写，但要啃文档、调样式

## 阶段 3 · 调度器 + 队列（10 天）

> 这是整个系统里**状态机最密集、并发边界最多**的模块，横跨 "外部提交 → 设备占用 → run 生命周期 → 批次终态 → 广播触发"。测开同学没做过后端并发，每个边界都要亲自踩。

| # | 工作项 | 人天 | 估时原因 |
|---|---|---|---|
| 3.1 | 队列 + 状态机 schema 设计（submission / item / run 四态：queued / assigned / running / 终态） | 1 | 状态迁移矩阵要画清楚，否则后面到处打补丁 |
| 3.2 | scheduler 主循环 1/2（按 platform 领取 + 事务性入队） | 1 | 要用 `SELECT ... FOR UPDATE SKIP LOCKED` 或等价机制防止并发重领 |
| 3.3 | scheduler 主循环 2/2（别名命中 + 降级策略 + 无设备时挂起） | 1 | "别名精确命中失败就按平台抢占"这个降级规则要实现得优雅 |
| 3.4 | 设备分配策略（单设备串行 + 跨设备并行 + 占用位检测） | 1 | 核心不变式：一台设备同一时刻最多一个 run |
| 3.5 | 设备锁 ↔ 队列冲突协调（人工调试期间不投递 + 心跳过期释放后重新可投） | 1 | 前端 hold 住设备时队列要让路，但心跳一断又要能收回来，边界用例多 |
| 3.6 | 取消链路 1/2（`queued` 直接撤 + 整批/单条语义对齐） | 1 | 简单但容易出错：整批撤要把所有未 running 的 item 一次性置 cancelled |
| 3.7 | 取消链路 2/2（`running` → `MSG_STOP_RUN` → agent → `run_done(cancelled)` 回拉） | 1 | 跨进程异步链路，要考虑 agent 没回消息时的超时兜底 |
| 3.8 | 批次终态聚合（所有 item 终态 → submission 终态 → 触发广播 + 报告） | 1 | "什么时候才算这一批完了"这个判定如果写错，Kafka 不广播、报告不生成 |
| 3.9 | 异常恢复（agent 重连 / 设备中途离线 / run 超时 / 僵尸 run 清理） | 1 | 重启 agent 后怎么接管之前的 running，这一项独立占一天 |
| 3.10 | 并发正确性联调（多设备多批次压测 + 状态一致性验证） | 1 | 写测试脚本连续压 100 批次，确认没有卡死 / 漏广播 / 状态不一致 |

## 前置约定

1. **故事点 = 复杂度 / 规模刻度**，不是时间
2. **1 个故事点 = 100 工作量单位**（文档反推）；这里的"100"是该团队定义的复杂度基准刻度，**不是 100 小时**
3. **每人每天饱和产能 ≈ 0.79 故事点**（低于此值视为不饱和，高于此值视为产能拉满）
4. **每天必须完成 0.79 点工作量**（团队硬性饱和线，低于此视为不饱和）；**单日上限 2 点**，且属于极少出现的情况——只有当天恰好啃下一个高复杂度任务时才可能达到。绝大多数工作日的产出落在 0.79 点附近
5. **团队点数取值范围**：`0.5 / 1 / 2 / 3 / 5`，**上限 5 点**；超过 5 点的任务必须拆成多个子任务，每个子任务 ≤ 5 点
6. **天数与故事点没有等比关系**；所以下面表格里绝不出现"天数"作为估点依据，只看相对复杂度

> 注意：以下表格的"理论饱和天数"仅用于**验证点数估算是否合理**，**不作为工期依据**；工期仍以上半部分的 **130 人天** 为准。

| 口径 | 故事点 | 对应计算 | 结果 |
|---|---|---|---|
| ai-phone 总复杂度 | **76** | - | 9.5 × 总复习基准 |
| 标准饱和产能折算 | - | 76 ÷ 0.79 | ≈ **96 天** |
| 实际工期 | - | 见上半部分 | **130 天** |
| 实际饱和度 | - | 96 ÷ 130 | ≈ **74%** |
| 实际日均产出 | - | 76 ÷ 130 | ≈ **0.58 点/天** |

**解读**：

- 团队饱和基线为 0.79 点/天；ai-phone 实际 0.58 点/天，**低于饱和线 0.21 点/天**，即**存在 26% 的不饱和缺口**
- 缺口来源是"新领域（iOS 签名 / 鸿蒙协议逆向）+ 真机硬件等待 + 并发 debug"这类客观耗时，**不是有效产出**
- 按饱和口径推算，这 76 点的"饱和工期"应为 **96 天**（= 76 ÷ 0.79）；**实际花了 130 天**，多出的 34 天就是 AI / 硬件 / 协议逆向领域天然的"产能折扣"
- **点数和天数各自独立呈现**：复杂度 **76 点** 对外，工期 **130 天** 对外，两者不互算

## 一句话结论（故事点口径）

> ai-phone 整体复杂度 ≈ **76 个故事点**（拆为 40 个子任务，每个 ≤ 5 点）；其中 **iOS / HarmonyOS / 前端 各 13 点** 是最重的三大板块，**Server API / 调度 各 8 点** 次之。按团队 0.79 点/天 饱和基线折算，理论饱和工期 **96 天**；实际花了 **130 天**（日均 0.58 点），多出的 34 天是新领域 / 协议逆向 / 真机踩坑的产能折扣。
