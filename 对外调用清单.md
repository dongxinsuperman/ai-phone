# 对外调用清单 v1

> 定位：**纯 AI 云真机执行器**对外 HTTP 契约（冻结版 v1）。
> 所有路径均以 server 基础地址为前缀，例：`http://localhost:5180`。
> 本文档与实现冻结在同一版本，字段/行为如果对不上以代码为准，并随时回来修文档。

## 适用场景

- 外部测试平台 / CI / 自动化脚本需要把一批自然语言测试指令投递到真机集群
- 异步执行、异步拿结果：轮询 GET 或订阅 Kafka 终态事件
- 想拿到每条 case 的可视化报告（HTML，可嵌入邮件 / Jenkins）

## 鉴权与权限模型

- **匿名**，不鉴权——按团队既定策略，靠网络隔离 / 防火墙圈外网访问
- 数据过期窗口：`submission` 终态后 **15 天**（由 `submission_external_retention_days` 控制）；
  过期后对外查询统一返回 `404 expired`，元数据和报告文件在服务端仍然保留

## 一览表

| Method | Path | 说明 |
| --- | --- | --- |
| GET  | `/api/devices/available` | 列出当前可接单设备（含 alias，未绑定 alias 为空串） |
| POST | `/api/submissions` | 投递一批 case |
| GET  | `/api/submissions/{submissionId}` | 查整批状态 + 所有 item 的 report_url |
| GET  | `/api/submissions/{submissionId}/items/{caseId}/{platform}` | 查单条详情（含 Run / Step / Log） |
| POST | `/api/submissions/{submissionId}/cancel` | 整批取消 |
| POST | `/api/submissions/{submissionId}/cases/{caseId}/cancel?platform=ios` | 单条取消 |

## 1. POST `/api/submissions` —— 投递

### 请求体

兼容两种顶层格式（**推荐用 wrapper 格式**）：

#### 格式 A：wrapper 对象（推荐，v1.3 引入）

```json
{
  "submissionName": "回归冒烟-2026-04-18 · 主流程",
  "items": [
    {
      "caseId": "smoke-login-001",
      "caseName": "注册 · 主流程",
      "platform": "ios",
      "runContent": "点击语文，点击物理",
      "deviceAlias": null
    },
    {
      "caseId": "smoke-login-001",
      "platform": "android",
      "runContent": "点击语文，点击物理"
    }
  ]
}
```

| 顶层字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `submissionName` | 可选 | string | 批次名称（=外部"集合 / 计划"概念）；展示用，缺省时各处展示回落 `submissionId` |
| `items` | ✅ | array | item 列表，结构同下表 |

#### 格式 B：纯数组（兼容老调用方）

```json
[
  {
    "caseId": "smoke-login-001",
    "caseName": "注册 · 主流程",
    "platform": "ios",
    "runContent": "点击语文，点击物理",
    "deviceAlias": null
  }
]
```

数组等价于 `{ "submissionName": "", "items": [...] }`，`submissionName` 为空时回落到 `submissionId`。

#### item 字段

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `caseId` | ✅ | string | 外部业务 case 标识；查询 / 取消都用它 |
| `caseName` | 可选 | string | 展示用；缺省时各处展示回落 `caseId` |
| `platform` | ⚠️ | string | **单端写法**。枚举：`android` / `ios` / `harmony`（大小写不敏感）。与 `platforms` **二选一** |
| `platforms`（v1.6 新增） | ⚠️ | string[] | **多端写法**。非空数组、元素不可重复、必须在枚举内。一条写法 B 的 item 会被服务端**展开成 N 条执行单元**（详见下节"单条风格与展开"）。与 `platform` **二选一** |
| `runContent` | ✅ | string | 自然语言指令，喂给 VLM |
| `deviceAlias` | 可选 | string | **单端配套**。严格别名：必须在 `device_aliases` 表里精确命中；缺省时调度器挑平台内任一 ready 设备。仅与 `platform`（单端）组合使用 |
| `deviceAliases`（v1.6 新增） | 可选 | object | **多端配套**：`{<platform>: <alias>}`。每个 key 必须出现在本条 `platforms` 里；值为 `null` / `""` 等同该端"不指定别名"（池内任挑）。仅与 `platforms`（多端）组合使用 |

#### 单条风格与展开

单条 item 内部**只能用一种风格**，禁止跨风格字段混用；**批次内不同 items 可以各自用不同风格**：

| 风格 | 必用字段 | 可选配套 | 生成 `SubmissionItem` 数 |
| --- | --- | --- | --- |
| A（单端，v1.0 起） | `platform`（string） | `deviceAlias`（string） | 1 |
| B（多端，v1.6 起） | `platforms`（array） | `deviceAliases`（object） | `len(platforms)` |

风格 B 展开规则：一条 `platforms: ["android", "ios", "harmony"]` 的 item 等价于
三条独立的 `SubmissionItem`（`case_id` 相同、`platform` 各异），走各自平台队列、
独立派发、独立终态、独立报告。响应 / 查询 / 取消都以展开后的每一条为准。

#### 规则

- 一批可以混平台（iOS + Android + HarmonyOS 一起投），每条独立走各自平台队列
- **批次内两种风格可以混用**：一部分 items 走风格 A（单端），另一部分走风格 B（多端）
- **单条内部字段组必须一致**：风格 A 不能出现 `platforms` / `deviceAliases`；风格 B
  不能出现 `platform` / `deviceAlias`。违反整批 400（具体 `rejectReason` 见下表）
- **同一 `(caseId, platform)` 不能在一批里出现两次**（含风格 B 展开后的结果），否则整批拒
- `deviceAlias` / `deviceAliases` 语义（v1.5 / v1.6 起）：
  - 任一别名不在 `device_aliases` 表里 → 整批 400 `unknown_device_alias`（不做 serial / model 前缀降级）
  - 别名命中但反查到的 serial 在 `devices` 表里的 platform 与 item 声明的 platform 不一致 → 整批 400 `device_alias_platform_mismatch`
    （例：`platform=android` 却传了绑在 iPhone 的别名；多端场景下 `deviceAliases: {"android": "<iOS别名>"}` 同样拒；直接在准入阶段拒，避免永远挂单到 submission 超时）
  - 别名指向的 serial 暂未上线（`devices` 表无记录）→ **不拒**，item 正常入队；等设备可用再按平台自然分流
  - 别名当前没 ready / 被占用 → **不拒**，item 正常入队；等设备可用就指名派发，3h 内还不行走 `queue_timeout`
  - 想知道"现在有哪些别名可选 + 每个别名对应什么平台" → 调 `GET /api/devices/available`
- 其它自定义字段被静默忽略（前向兼容保留口）

#### 风格 B 示例（一条 case 跑三端 + 两端指名、一端随机）

```json
{
  "submissionName": "三端冒烟",
  "items": [
    {
      "caseId": "smoke-login-001",
      "caseName": "注册 · 主流程",
      "runContent": "注册一个账号并完成手机绑定",
      "platforms": ["android", "ios", "harmony"],
      "deviceAliases": {
        "android": "dx-android-01",
        "ios": "dx-iphone-03"
      }
    }
  ]
}
```

服务端会展开成 3 条 `SubmissionItem`（android / ios / harmony 各一条）；`harmony`
未在 `deviceAliases` 里列出，该条走池内任挑。

### 成功响应 `201 Created`

```json
{
  "submissionId": "abcdef012345",
  "submissionName": "回归冒烟-2026-04-18 · 主流程",
  "acceptedAt": "2026-04-22T03:14:15+00:00",
  "expireAt": "2026-04-22T06:14:15+00:00",
  "items": [
    {
      "itemId": 1001,
      "caseId": "smoke-login-001",
      "caseName": "注册 · 主流程",
      "platform": "ios",
      "deviceAlias": null,
      "state": "queued"
    }
  ]
}
```

拿到 `submissionId` 之后，所有查询 / 取消都以它为外部主键。`submissionName` 仅为展示
位（报告标题、Web 队列页、Kafka 终态事件），不参与路由或唯一性约束；缺省 / 留空时
回落到 `submissionId`。

### 失败响应 `400 Bad Request`

```json
{
  "detail": {
    "rejectReason": "invalid_platform",
    "rejectDetail": "platform 必须是 android / ios / harmony，收到 'Ios'",
    "index": 0
  }
}
```

`rejectReason` 枚举：

| 值 | 含义 |
| --- | --- |
| `invalid_body` | 请求体既不是 JSON 数组也不是 wrapper 对象 / `items` 为空 / 某条不是 object / `platforms` 不是非空数组 / `platforms` 内有重复 / `deviceAliases` 不是对象 |
| `missing_field` | `caseId` / `runContent` 缺失，或 `platform` 和 `platforms` 都没传 |
| `invalid_platform` | `platform` 或 `platforms` 里的元素不在枚举内；或 `deviceAliases` 的 key 不是合法平台 |
| `conflicting_platform_field`（v1.6） | 一条 item 里同时传了 `platform` 和 `platforms` |
| `conflicting_alias_field`（v1.6） | 一条 item 里同时传了 `deviceAlias` 和 `deviceAliases` |
| `mixed_field_style`（v1.6） | 单条内字段组跨风格错配：`platforms` 配 `deviceAlias`（该改 `deviceAliases`），或 `platform` 配 `deviceAliases`（该改 `deviceAlias`） |
| `alias_platform_not_in_list`（v1.6） | `deviceAliases` 里某个平台 key 不在本条 `platforms` 里 |
| `no_device_on_platform` | 某平台当前没有任何 online 设备（风格 B 展开后的任一端没 online 设备也一样拒） |
| `unknown_device_alias` | `deviceAlias` / `deviceAliases` 中的别名未在 `device_aliases` 表里命中（`rejectDetail` 里带全部未命中的 alias） |
| `device_alias_platform_mismatch` | 别名命中了，但反查到的 serial 对应设备不是 item 声明的 platform（v1.5 起）。`rejectDetail` 带全部冲突项，格式：`<alias>(绑定设备 serial=<serial> 平台=<actual>, 与 item 声明 platform=<declared> 不一致)`。先绑后现场景（serial 尚未上线）不会命中本错误 |

> 同批 `(caseId, platform)` 去重规则计划表里写过，但 v1 执行器侧**暂未强制校验**——调用方侧一般也不会把重复项打进一批。后续如果正式启用，会在这里补 `duplicate_in_batch` 并同步升级版本号。

## 2. GET `/api/submissions/{submissionId}` —— 查整批

### 响应

```json
{
  "id": "abcdef012345",
  "submission_id": "abcdef012345",
  "submission_name": "回归冒烟-2026-04-18 · 主流程",
  "state": "running",
  "origin": "external",
  "accepted_at": "2026-04-22T03:14:15+00:00",
  "expire_at": "2026-04-22T06:14:15+00:00",
  "finished_at": null,
  "summary_report_url": null,
  "items": [
    {
      "id": 1001,
      "case_id": "smoke-login-001",
      "case_name": "注册 · 主流程",
      "platform": "ios",
      "device_alias": null,
      "state": "success",
      "run_id": "c7bc99a8353a",
      "enqueued_at": "2026-04-22T03:14:15+00:00",
      "started_at": "2026-04-22T03:14:20+00:00",
      "finished_at": "2026-04-22T03:14:38+00:00",
      "status_reason": null,
      "report_url": "http://localhost:5180/files/reports/abcdef012345/1001__ios.html"
    }
  ],
  "counts": { "queued": 0, "running": 1, "success": 1 },
  "external_retention_days": 15
}
```

### 状态枚举

**`submission.state`**：`accepted` → `running` → `success` / `failed` / `cancelled`

**`item.state`**：`queued` → `running` → `success` / `failed` / `cancelled`

**`item.status_reason`** 枚举（v1 共 **11 项**；`success` 时固定 `completed`，`failed` / `cancelled` 时从其余 10 项中取一）：

| 值 | result | 归类 | 含义 / 触发时机 |
| --- | --- | --- | --- |
| `completed` | success | — | 正常完成（VLM `finished` 或等价成功条件）|
| `assert_failed` | fail | 业务 | 模型主动下发 `assert_fail` 动作 |
| `run_timeout` | fail | 平台 | item 进入 `running` 后超 1h 未终态 |
| `queue_timeout` | fail | 平台 | 预留；v1 靠 submission 3h 兜底，暂不细分 |
| `submission_timeout` | fail | 平台 | submission 到 3h 硬上限时，仍在 `queued` 的 item 逐条被踢；**某平台所有设备长期离线 / 抖动收不回来，最终也走这个**（v1 不做"平台全灭快踢"，避免抖动错杀队列）|
| `cancelled_by_request` | fail | 人为 | 外部调取消接口命中 `queued` item |
| `stuck_detected` | fail | 平台 | 卡死保护（点击/滚动连续 N 次无效果）|
| `vlm_unavailable` | fail | 平台 | VLM API 超时 / 5xx / 鉴权失败等外部依赖故障 |
| `device_unavailable` | fail | 平台 | 派单时目标设备不 ready（锁屏 / WDA 未就绪 / driver 探测失败等）|
| `executor_resource_lost` | fail | 平台 | `running` 中设备掉线 / agent 断链 |
| `executor_error` | fail | 平台 | 执行器内部异常、未分类错误兜底 |

> **消费方使用建议**：`result = fail` 时先看 `statusReason` 是否 ∈ 业务/人为集合（`assert_failed` / `cancelled_by_request`），是的话归"业务/人为失败"、不进稳定性分母；其余视作"平台失败"，自行决定要不要再发一笔新 submission 重跑。执行器侧**不会自动重跑**。

> **关于"平台所有设备同时离线"**：v1 不把这种情况做成独立枚举，因为 agent 抖动 / USB 瞬断 / 设备拔插是常态，一旦设置独立"快踢"路径，一次抖动可能把整条队列打飞。保守做法就是 queued 继续等机器回来，等不到再由 submission 3h 超时统一以 `submission_timeout` 收口。

### 过期响应 `404 Not Found`

```json
{
  "detail": {
    "error": "expired",
    "message": "submission 已超过对外可查窗口（15 天），元数据与报告仍在服务端保留，但对外不再开放查询"
  }
}
```

## 4. POST `/api/submissions/{submissionId}/cancel` —— 整批取消

无请求体。

### 响应

```json
{
  "submissionId": "abcdef012345",
  "cancelled_items": [1001, 1002],
  "skipped_items": []
}
```

规则：

- `queued` / `running` 的 item 会被打断
- 已终态的 item 保留原结果，进入 `skipped_items`
- 全部 item 收敛终态后，`submission.state` 收敛到 `cancelled`

## 报告 URL 约定

每条 item 进入终态（`success` / `failed`）后，响应里的 `report_url` 就有值。

- 单 case 报告：`http://{server}/files/reports/{submissionId}/{itemId}__{platform}.html`
- 批次汇总报告（所有 item 终态后）：`http://{server}/files/reports/{submissionId}/_summary.html`

报告是**自包含 HTML**：

- 内联 CSS / JS，图片走服务端图床 URL
- 可以邮件转发、CI 归档、Jenkins / 飞书卡片直接嵌入
- 单 case 采用时间线视图（log 和 step 按时间戳归并）
- 批次汇总是 SPA：左侧用例列表，右侧复用单 case 模板，`caseName` 为主显示
- 点击任意截图可放大查看

## 和内部 API 的区别

仅用于接入方心里有数，对外不暴露。

| 方面 | 内部 `/api/internal/submissions` | 外部 `/api/submissions`（本文档） |
| --- | --- | --- |
| 鉴权 | `Authorization: Bearer` | 匿名 |
| `origin` 字段 | `internal` | `external` |
| 15 天过期 | 不关心 | 过期后统一 `404 expired` |

两者共享同一个 `SubmissionScheduler` 实例和同一份 `Submission` / `SubmissionItem` 表，不会串扰。
