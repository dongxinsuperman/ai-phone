# 对外调用清单 v1

> 定位：**纯 AI 云真机执行器**对外 HTTP 契约（冻结版 v1）。
> 所有路径均以 server 基础地址为前缀，例：`http://localhost:5180`。
> 本文档与实现冻结在同一版本，字段/行为如果对不上以代码为准，并随时回来修文档。

## 适用场景

- 外部测试平台 / CI / 自动化脚本需要把一批自然语言测试指令投递到真机集群
- 异步执行、异步拿结果：轮询 GET 或订阅 Kafka 终态事件
- 想拿到每条 case 的可视化报告（HTML，可嵌入邮件 / Jenkins）

## 鉴权与权限模型

- **匿名**，不鉴权——按团队既定策略，靠网络隔离 / 防火墙圈外网访问
- 数据过期窗口：`submission` 终态后 **15 天**（由 `submission_external_retention_days` 控制）；
  过期后对外查询统一返回 `404 expired`，元数据和报告文件在服务端仍然保留

## 一览表

| Method | Path | 说明 |
| --- | --- | --- |
| GET  | `/api/devices/available` | 列出当前可接单设备（含 alias，未绑定 alias 为空串） |
| POST | `/api/submissions` | 投递一批 case |
| GET  | `/api/submissions/{submissionId}` | 查整批状态 + 所有 item 的 report_url |
| GET  | `/api/submissions/{submissionId}/items/{caseId}/{platform}` | 查单条详情（含 Run / Step / Log） |
| POST | `/api/submissions/{submissionId}/cancel` | 整批取消 |
| POST | `/api/submissions/{submissionId}/cases/{caseId}/cancel?platform=ios` | 单条取消 |
| Kafka | topic `ai-phone.submission.result` | 终态事件流（item 终态 + 批次收口），见 §6 |
| Webhook | 投递时传 `callbackUrl` | 整批收口时 ai-phone 异步 POST 一次回调，per-batch 跟随，见 §7 |

## 0. GET `/api/devices/available` —— 查可用设备（含别名）

**匿名**；外部调用方在投递前通常先问一下"当前有哪些设备能接，它们的别名分别是什么"。
返回当下**真正能立刻派单**的设备（`online` + readiness ready + 无锁 + agent 在线）。

> 注意：没绑别名的设备照样会出现在列表里，`alias` 字段为空串；这种设备只能走"不指定
> `deviceAliasPools`"的随机派发，不能被外部精确指名。

### 响应

```json
[
  {
    "serial": "00008150-00041CAE3478401C",
    "alias": "ios-iphone14-shanghai-01",
    "platform": "ios",
    "brand": "Apple",
    "model": "iPhone 14",
    "osVersion": "17.5.1",
    "screenWidth": 1170,
    "screenHeight": 2532,
    "lastSeenAt": "2026-04-22T03:14:15+00:00"
  },
  {
    "serial": "6ad9243",
    "alias": "",
    "platform": "android",
    "brand": "Google",
    "model": "Pixel 7",
    "osVersion": "14",
    "screenWidth": 1080,
    "screenHeight": 2400,
    "lastSeenAt": "2026-04-22T03:14:12+00:00"
  }
]
```

### 典型用法

```bash
# 拿可用设备清单
curl -s http://localhost:5180/api/devices/available | jq

# 根据 alias 投递到指定设备
curl -s -X POST http://localhost:5180/api/submissions \
  -H "Content-Type: application/json" \
  -d '{
    "submissionName": "定点回归",
    "items": [
      {"caseId":"t1","platforms":["ios"],"runContent":"点击语文",
       "deviceAliasPools":{"ios":["ios-iphone14-shanghai-01"]}}
    ]
  }'
```

## 1. POST `/api/submissions` —— 投递

### 请求体（v1.7 唯一形态）

```json
{
  "submissionName": "回归冒烟-2026-04-18 · 主流程",
  "items": [
    {
      "caseId": "smoke-login-001",
      "caseName": "注册 · 主流程",
      "runContent": "点击语文，点击物理",
      "platforms": ["ios"]
    },
    {
      "caseId": "smoke-login-002",
      "runContent": "打开商城下单",
      "platforms": ["android"],
      "deviceAliasPools": {"android": ["dx-android-01", "dx-android-02"]}
    }
  ]
}
```

> v1.6 及以前的纯数组顶层、单端 `platform` / `deviceAlias`、多端 `deviceAliases`（字典）
> 字段都已在 v1.7 **彻底废弃**——传这些字段会被当成 `invalid_body` 拒。

#### 顶层字段

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `submissionName` | 可选 | string | 批次名称（=外部"集合 / 计划"概念）；展示用，缺省时各处展示回落 `submissionId` |
| `callbackUrl` | 可选 | string | **v1.8 webhook**：本批整批跑完时，ai-phone 异步 POST 一份 `submission.terminal` payload 到这个 URL（与 Kafka 通道并存）。必须 `http://` 或 `https://` 开头，长度 ≤ 1024。失败发一次就吞，不重试不签名；需要可靠投递请走 Kafka。详见 §7 |
| `cacheMode` | 可选 | string | **v1.9 缓存策略**：本批所有 item 共用的缓存模式，枚举 `"off"` / `"v1"` / `"v2"` / `"v3"`，缺省 `"off"`。**全静默对齐**：服务端总开关关 / 枚举外的值（拼写错、null、其他类型）一律按 `"off"` 执行 + INFO 日志，不返回 400。详见 §1.x「缓存与重跑」 |
| `retryMax` | 可选 | int | **v1.9 自动重跑**：本批所有 item 在失败时的重跑次数（同 Run 内复跑），≥ 0 整数，缺省 `0`。**全静默对齐**：服务端总开关关 / 上限不足 / 类型错误 / 负数 一律截断到允许范围 + INFO 日志，不返回 400。详见 §1.x「缓存与重跑」 |
| `items` | ✅ | array | item 列表，至少 1 条；结构同下表 |

#### item 字段

| 字段 | 必填 | 类型 | 说明 |
| --- | --- | --- | --- |
| `caseId` | ✅ | string | 外部业务 case 标识；查询 / 取消都用它 |
| `caseName` | 可选 | string | 展示用；缺省时各处展示回落 `caseId` |
| `runContent` | ✅ | string | 自然语言指令，喂给 VLM |
| `platforms` | ✅ | string[] | 非空数组，元素 ∈ {`android`, `ios`, `harmony`}（大小写不敏感），不可重复。**一条 item 会按 `platforms` 展开成 `len(platforms)` 条执行单元**，每个 platform 各占一行 `SubmissionItem`，走各自平台队列、独立派发、独立报告 |
| `deviceAliasPools` | 可选 | object | **按端别名池**：`{<platform>: <string[]>}`。每个 key 必须出现在本条 `platforms` 里。三档语义见下表 |

#### `deviceAliasPools` 三档语义

| 写法 | 含义 |
| --- | --- |
| 字段缺省 / 该端 key 缺失 / 值为 `null` / 值为 `[]` | **全池任挑**——该端任意 ready 设备 |
| 长度 1（`["A1"]`）| **锁单台**——只在这台上跑 |
| 长度 N（`["A1","B1",...]`）| **子集池**——调度器在派发瞬间挑池里 ready 的一台。哪台先 ready 就被哪台拿走，自然形成"快机多跑、慢机少跑、坏机不跑"的负载分担 |

子集池是 v1.7 的核心新能力——多业务方"资产私有 + 平台共有"场景下，业务方可以
把自己团队的多台 alias 一起填进 pool，平台只在这几台之间动态分配，不会跑到
别的团队设备上。

#### 规则

- 一批可以混平台（iOS + Android + HarmonyOS 一起投），每条独立走各自平台队列
- **同一 `(caseId, platform)` 在同一批里只能出现一次**（含 `platforms` 展开后的结果），否则整批拒
  - 想做"同 case 在同端多台机器各跑一遍"的兼容性测试，请提交多个 submission，每个 pool 锁不同的单台
- 别名校验（v1.7）：
  - 池里**任一别名**不在 `device_aliases` 表里 → 整批 400 `unknown_device_alias`
  - 池里别名命中但反查到的 serial 在 `devices` 表里的 platform 与 item 声明的 platform 不一致 → 整批 400 `device_alias_platform_mismatch`
  - 别名指向的 serial 暂未上线（`devices` 表无记录）→ **不拒**，item 正常入队；等设备可用再按平台自然分流
  - 别名当前没 ready / 被占用 → **不拒**，item 正常入队；等设备可用就动态指名派发，3h 内还不行走 `queue_timeout`
  - 想知道"现在有哪些别名可选 + 每个别名对应什么平台" → 调 `GET /api/devices/available`
- 其它自定义字段被静默忽略（前向兼容保留口）

#### 多端 + 子集池综合示例

```json
{
  "submissionName": "三端冒烟 · 各端两台共池",
  "items": [
    {
      "caseId": "smoke-login-001",
      "caseName": "注册 · 主流程",
      "runContent": "注册一个账号并完成手机绑定",
      "platforms": ["android", "ios", "harmony"],
      "deviceAliasPools": {
        "android": ["dx-android-01", "dx-android-02"],
        "ios": ["dx-iphone-03"]
      }
    }
  ]
}
```

服务端展开成 3 条 `SubmissionItem`：
- **android** → 池 `[dx-android-01, dx-android-02]`：哪台先 ready 谁拿走（子集池）
- **ios** → 池 `[dx-iphone-03]`：锁这一台
- **harmony** → 池缺省：该端任意 ready 设备

#### 缓存与重跑（v1.9 新增 · 顶层批次级策略）

`cacheMode` 与 `retryMax` 是 v1.9 引入的两个**顶层**可选字段，**作用于本批所有 item**——
服务端在派发时自动注入到每个 `SubmissionItem` 派发的 Run 里，**不允许 item 级单独覆盖**。
两个字段相互独立，可任意组合。

##### 取值与对齐规则

| 字段 | 类型 | 取值 | 缺省 | 越界/非法处理 |
| --- | --- | --- | --- | --- |
| `cacheMode` | string | `"off"` / `"v1"` / `"v2"` / `"v3"` | `"off"` | 枚举外（拼写错、其他类型）→ 静默 `"off"` + INFO 日志 |
| `retryMax` | int | ≥ 0 | `0` | 类型错 / 负数 → 静默 `0`；超过服务端上限 → 截断到上限。均配 INFO 日志 |

> **设计原则**：用最大的容忍接收，用最稳的行为执行。任何配置不一致都对齐到最安全的"不缓存 / 不重跑"，
> Run 仍然能跑（只是不享受加速 / 重跑保护），不会因为这两个字段把整批 400 打回。

##### 服务端总开关 = 能力闸门

服务端可独立关闭这两类能力（运维侧 env 配置）：
- 缓存总开关关 → 任何 `cacheMode` 都按 `off` 执行
- 重跑总开关关 / 上限为 0 → 任何 `retryMax` 都按 `0` 执行
- 服务端能力关闭时**不会拒绝投递**，只在 Run 日志里 INFO 一行说明本次按安全值降级

##### 四种典型 payload

```jsonc
// 场景 1：什么都不要（默认行为，不缓存 / 不重跑）
// 等价于不传 cacheMode / retryMax
{
  "submissionName": "纯 VLM 跑",
  "items": [{"caseId": "c-001", "runContent": "...", "platforms": ["ios"]}]
}

// 场景 2：只要 V1 极速缓存
{
  "submissionName": "回归冒烟 · V1 极速",
  "cacheMode": "v1",
  "items": [{"caseId": "c-001", "runContent": "...", "platforms": ["ios"]}]
}

// 场景 3：只要重跑（不缓存）—— 抗瞬时网络抖动
{
  "submissionName": "稳定性巡检 · 重跑兜底",
  "retryMax": 2,
  "items": [{"caseId": "c-001", "runContent": "...", "platforms": ["android"]}]
}

// 场景 4：V3 在线识别 + 重跑 1 次（最强组合，最慢）
{
  "submissionName": "UI 漂移压测 · V3 + 重跑",
  "cacheMode": "v3",
  "retryMax": 1,
  "items": [{"caseId": "c-001", "runContent": "...", "platforms": ["ios", "android"]}]
}
```

##### 跟单条调试入口（`POST /api/runs`）的对齐

服务端为了"投递 / 调试"两条入口的语义统一，`POST /api/runs` 也接受同名同语义的
`cacheMode` / `retryMax` 顶层字段，缺省同样是 `"off"` / `0`，不再单独说明。
对于本文档的对外用户来说，只需要关心 `POST /api/submissions` 的用法。

### 成功响应 `201 Created`

```json
{
  "submissionId": "abcdef012345",
  "submissionName": "回归冒烟-2026-04-18 · 主流程",
  "acceptedAt": "2026-04-22T03:14:15+00:00",
  "expireAt": "2026-04-22T06:14:15+00:00",
  "items": [
    {
      "itemId": 1001,
      "caseId": "smoke-login-001",
      "caseName": "注册 · 主流程",
      "platform": "ios",
      "deviceAliasPool": null,
      "state": "queued"
    }
  ]
}
```

拿到 `submissionId` 之后，所有查询 / 取消都以它为外部主键。`submissionName` 仅为展示
位（报告标题、Web 队列页、Kafka 终态事件），不参与路由或唯一性约束；缺省 / 留空时
回落到 `submissionId`。

### 失败响应 `400 Bad Request`

```json
{
  "detail": {
    "rejectReason": "invalid_platform",
    "rejectDetail": "platform 必须是 android / ios / harmony，收到 'Ios'",
    "index": 0
  }
}
```

`rejectReason` 枚举（v1.7 精简）：

| 值 | 含义 |
| --- | --- |
| `invalid_body` | 请求体不是 wrapper 对象 / `items` 不是数组 / `items` 为空 / 某条不是 object / `platforms` 不是非空数组 / `platforms` 内有重复 / `deviceAliasPools` 不是对象 / `deviceAliasPools[p]` 不是数组或 null / 池里含空别名 |
| `missing_field` | `caseId` / `runContent` / `platforms` 缺失 |
| `invalid_platform` | `platforms` 里的元素不在枚举内；或 `deviceAliasPools` 的 key 不是合法平台 |
| `pool_alias_not_in_platforms` | `deviceAliasPools` 里某个平台 key 不在本条 `platforms` 里 |
| `no_device_on_platform` | 某平台当前没有任何 online 设备 |
| `unknown_device_alias` | `deviceAliasPools` 中的某别名未在 `device_aliases` 表里命中（`rejectDetail` 里带全部未命中的 alias） |
| `device_alias_platform_mismatch` | 池里别名命中了，但反查到的 serial 对应设备不是 item 声明的 platform。`rejectDetail` 带全部冲突项。先绑后现场景（serial 尚未上线）不会命中本错误 |

> 同批 `(caseId, platform)` 去重规则计划表里写过，但 v1 执行器侧**暂未强制校验**——调用方侧一般也不会把重复项打进一批。后续如果正式启用，会在这里补 `duplicate_in_batch` 并同步升级版本号。

## 2. GET `/api/submissions/{submissionId}` —— 查整批

### 响应

```json
{
  "id": "abcdef012345",
  "submission_id": "abcdef012345",
  "submission_name": "回归冒烟-2026-04-18 · 主流程",
  "state": "running",
  "origin": "external",
  "accepted_at": "2026-04-22T03:14:15+00:00",
  "expire_at": "2026-04-22T06:14:15+00:00",
  "finished_at": null,
  "summary_report_url": null,

  // v1.9 新增：批次策略字段
  // 三组都是"决议后"的实际值；如果用户传了 cacheMode/retryMax 但被服务端降级，
  // requested_* 保留原值便于排查，effective_* 是真正生效值
  "requested_cache_mode": "v3",
  "effective_cache_mode": "v3",
  "requested_retry_max": 99,
  "effective_retry_max": 2,

  "items": [
    {
      "id": 1001,
      "case_id": "smoke-login-001",
      "case_name": "注册 · 主流程",
      "platform": "ios",
      "device_alias_pool": null,
      "state": "success",
      "run_id": "c7bc99a8353a",
      "enqueued_at": "2026-04-22T03:14:15+00:00",
      "started_at": "2026-04-22T03:14:20+00:00",
      "finished_at": "2026-04-22T03:14:38+00:00",
      "status_reason": null,
      "report_url": "http://localhost:5180/files/reports/abcdef012345/1001__ios.html",

      // v1.9 新增：本 item 实际跑了几次（1 = 没重跑过；>1 = 触发了 retry）
      "attempts": 2
    }
  ],
  "counts": { "queued": 0, "running": 1, "success": 1 },
  "external_retention_days": 15
}
```

#### v1.9 新增字段说明

| 字段 | 层级 | 类型 | 含义 |
| --- | --- | --- | --- |
| `requested_cache_mode` | submission | string | 用户投递时传的原始 `cacheMode`，可能是 `null`（未传）或非法值（如 `"V2"`）|
| `effective_cache_mode` | submission | string | 决议后实际生效，必为 `"off"`/`"v1"`/`"v2"`/`"v3"` 之一 |
| `requested_retry_max` | submission | int \| null | 用户投递时传的原始 `retryMax`，未传为 `null` |
| `effective_retry_max` | submission | int | 决议后实际生效，0 ≤ `effective_retry_max` ≤ 服务端 env 上限 |
| `attempts` | item | int | 本 item 实际跑了几次。`1` = 一次成功 / 无重跑；`>1` = 触发了 retry。无论最终 success/fail 都填实际次数 |

### 状态枚举

**`submission.state`**：`accepted` → `running` → `success` / `failed` / `cancelled`

**`item.state`**：`queued` → `running` → `success` / `failed` / `cancelled`

**`item.status_reason`** 枚举（v1 共 **11 项**；`success` 时固定 `completed`，`failed` / `cancelled` 时从其余 10 项中取一）：

| 值 | result | 归类 | 含义 / 触发时机 |
| --- | --- | --- | --- |
| `completed` | success | — | 正常完成（VLM `finished` 或等价成功条件）|
| `assert_failed` | fail | 业务 | 模型主动下发 `assert_fail` 动作 |
| `run_timeout` | fail | 平台 | item 进入 `running` 后超 1h 未终态 |
| `queue_timeout` | fail | 平台 | 预留；v1 靠 submission 3h 兜底，暂不细分 |
| `submission_timeout` | fail | 平台 | submission 到 3h 硬上限时，仍在 `queued` 的 item 逐条被踢；**某平台所有设备长期离线 / 抖动收不回来，最终也走这个**（v1 不做"平台全灭快踢"，避免抖动错杀队列）|
| `cancelled_by_request` | fail | 人为 | 外部调取消接口命中 `queued` item |
| `stuck_detected` | fail | 平台 | 卡死保护（点击/滚动连续 N 次无效果）|
| `vlm_unavailable` | fail | 平台 | VLM API 超时 / 5xx / 鉴权失败等外部依赖故障 |
| `device_unavailable` | fail | 平台 | 派单时目标设备不 ready（锁屏 / WDA 未就绪 / driver 探测失败等）|
| `executor_resource_lost` | fail | 平台 | `running` 中设备掉线 / agent 断链 |
| `executor_error` | fail | 平台 | 执行器内部异常、未分类错误兜底 |

> **消费方使用建议**：`result = fail` 时先看 `statusReason` 是否 ∈ 业务/人为集合（`assert_failed` / `cancelled_by_request`），是的话归"业务/人为失败"、不进稳定性分母；其余视作"平台失败"，自行决定要不要再发一笔新 submission 重跑。执行器侧**不会自动重跑**。

> **关于"平台所有设备同时离线"**：v1 不把这种情况做成独立枚举，因为 agent 抖动 / USB 瞬断 / 设备拔插是常态，一旦设置独立"快踢"路径，一次抖动可能把整条队列打飞。保守做法就是 queued 继续等机器回来，等不到再由 submission 3h 超时统一以 `submission_timeout` 收口。

### 过期响应 `404 Not Found`

```json
{

    "error": "expired",
    "message": "submission 已超过对外可查窗口（15 天），元数据与报告仍在服务端保留，但对外不再开放查询"
```

- 全部 item 收敛终态后，`submission.state` 收敛到 `done` 或 `cancelled`（取决于整批取消路径和已完成 item 的组合）

## 6. Kafka 异步广播（订阅终态结果）

除了轮询 `GET /api/submissions/{id}`，调用方还可以订阅 Kafka topic 拿到**实时**终态事件。

### Topic / 顺序契约

| 项 | 值 | 说明 |
| --- | --- | --- |
| Topic | `ai-phone.submission.result` | v1 固定；多环境隔离请走多 broker 而非多 topic |
| 分区键 | `submissionId`（UTF-8 字符串） | **同一批次的所有事件落同一分区** → 同分区按 offset 严格有序消费 |
| value 编码 | UTF-8 JSON | `application/json`，字段名小驼峰 |
| 投递语义 | at-least-once | producer `acks=all` + idempotent；消费方需自行去重（推荐 `submissionId + itemId + event` 三元组做幂等键） |
| 广播范围 | **仅终态** | `queued` / `running` 中间态不广播 |

### 两种事件类型

同一 topic 混合两种事件，调用方按 payload 里的 `event` 字段路由：

| `event` | 触发时机 | 频率 |
| --- | --- | --- |
| `submission.item.terminal` | 单条 item（一个 caseId × 一端）跑到终态 | 每批 N 条（N = `platforms` 展开后的 item 总数） |
| `submission.terminal` | 整批所有 item 都终态后**收口** | 每批 1 条；**永远在所有 item.terminal 之后**到达 |

同一批次的事件序列示例（按 offset 顺序）：

```
offset=0  event=submission.item.terminal   case A android success
offset=1  event=submission.item.terminal   case B ios     failed
offset=2  event=submission.item.terminal   case C android success
offset=3  event=submission.terminal        批次收口（counts/聚合）
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `event` | string | 固定 `submission.item.terminal`，调用方靠它路由 |

| `platform` | string | `android` / `ios` / `harmony` |
| `engine` | string \| null | 执行引擎 `vlm` / `midscene`；item 在 queued 阶段被收尾时为 `null`（没绑 Run） |
| `state` | string | `success` / `failed` / `cancelled` |
| `statusReason` | string \| null | 11 项枚举之一，见 §2 状态枚举表 |
| `runId` | string \| null | 实际派发到的 Run ID；queued 阶段被收尾时为 `null` |

| `finishedAt` | ISO8601 \| null | 终态落位时刻；未启动时为 `null` |
| `elapsedMs` | int \| null | `finishedAt - startedAt`；未启动时为 `null`。**包含所有 attempt 的总耗时** |
| `steps` | int \| null | 执行步数（来自 Run，最后一次 attempt 的）；queued 收尾时为 `null` |
| `tokenStats` | object | LLM token 摘要，**跨 attempt 累加**；queued 收尾时为 `{}` |
| `reportUrl` | string \| null | HTML 报告地址；queued 阶段被收尾时为 `null`（无报告） |
| `origin` | string | `external`（对外 API 投的） / `internal`（平台内部投的） |

### 事件 2：`submission.terminal`（整批收口）

```json
{
  "event": "submission.terminal",
  "version": 1,
  "ts": "2026-04-30T03:14:16.000+00:00",

  "submissionId": "abcdef012345",
  "submissionName": "回归冒烟-2026-04-18 · 主流程",
  "origin": "external",
  "submissionState": "done",

  "acceptedAt": "2026-04-30T03:13:45.884+00:00",
  "finishedAt": "2026-04-30T03:14:15.884+00:00",
  "totalItems": 10,

  "counts":         {"success": 9, "failed": 1},
  "platformCounts": {"android": 5, "ios": 5},
  "platformStateCounts": {
    "android": {"success": 4, "failed": 1},
    "ios":     {"success": 5}
  },
  "totalElapsedMs": 200000,

  "summaryReportUrl": "/files/reports/abcdef012345/_summary.html",

  // v1.9 新增：批次策略与 retry 聚合
  "cacheMode": "v3",       // 本批实际生效的 cacheMode（来自 submission 顶层决议）
  "retryMax": 1,           // 本批实际生效的 retryMax 上限
  "retryStats": {           // 本批 retry 触发情况摘要（聚合所有 item）
    "itemsWithRetry": 3,    //   有过 retry 的 item 数（attempts > 1）
    "totalAttempts": 14,    //   所有 item 的 attempts 之和
    "successAfterRetry": 2  //   重跑后成功的 item 数（其中 attempts > 1 且 final state = success）
  }
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `submissionState` | string | `done` / `cancelled` / `expired`；`accepted` 不会广播（还没收口） |
| `totalItems` | int | 本批 `platforms` 展开后的 item 总数 |
| `counts` | object | 按 state 聚合：`{state: count}` |
| `platformCounts` | object | 按 platform 聚合：`{platform: count}` |
| `platformStateCounts` | object | **端 × 状态二维矩阵**：`{platform: {state: count}}`，业务方一眼看出每端合格率 |
| `totalElapsedMs` | int | 所有 item `(finishedAt-startedAt)` 总和（含 retry 累计耗时）；只统计真正跑过的 item |
| `summaryReportUrl` | string \| null | 批次汇总 HTML；生成失败时为 `null`，调用方降级走 JSON 接口 |
| `cacheMode` | string | **v1.9 新增**：本批实际生效的 cacheMode |
| `retryMax` | int | **v1.9 新增**：本批实际生效的 retryMax 上限 |
| `retryStats.itemsWithRetry` | int | **v1.9 新增**：本批触发过 retry 的 item 数（`attempts > 1`） |
| `retryStats.totalAttempts` | int | **v1.9 新增**：所有 item `attempts` 字段之和（看资源消耗倍数） |
| `retryStats.successAfterRetry` | int | **v1.9 新增**：靠 retry 救回来的 item 数（重跑后最终 success），用于评估 retry 价值 |

### 边界场景下广播什么

| 场景 | item.terminal | submission.terminal |
| --- | --- | --- |
| 整批 `/cancel`（部分 queued） | queued 项发 `state=cancelled` + `statusReason=cancelled_by_request`，`reportUrl=null`；running 项继续跑自己 | submissionState=`cancelled` |
| 单条 `/cancel`（queued） | 该项 `state=cancelled`；其它项不受影响 | 等所有 item 收齐后正常发 |
| 批次 `submission_timeout`（3h 硬上限） | 仍 queued 的项 `state=cancelled` + `statusReason=submission_timeout` | submissionState=`expired` |
| Agent 短暂掉线 | 不广播；item 留在 running 等 agent 回来，`status_reason=agent_lost` 才会终态 | 同上 |

> **重要保证**：`submission.terminal` 一定在所有该批次的 `submission.item.terminal` 之后到达（同分区有序）。调用方等到 `submission.terminal` 即可认为本批次彻底收口。

### 消费方最小代码示例（Python / aiokafka）

```python
import json
from aiokafka import AIOKafkaConsumer

async def consume():
    consumer = AIOKafkaConsumer(
        "ai-phone.submission.result",
        bootstrap_servers="kafka-host:9092",
        group_id="my-app-group",        # 同 group 多实例自动负载均衡
        auto_offset_reset="latest",     # 历史不补，只接新消息
        enable_auto_commit=False,       # 业务处理成功后再 commit，配合幂等去重
    )
    await consumer.start()
    try:
        async for msg in consumer:
            sub_id = msg.key.decode()                     # = submissionId
            body = json.loads(msg.value)
            et = body["event"]
            if et == "submission.item.terminal":
                handle_item(body)            # 单条 case 终态
            elif et == "submission.terminal":
                handle_batch_summary(body)   # 整批收口
            else:
                pass                          # 前向兼容：未知事件类型忽略
            await consumer.commit()
    finally:
        await consumer.stop()
```

幂等去重建议：用 `(submissionId, itemId, event)` 三元组做 dedup key（item 事件用 itemId，submission 事件 itemId 不存在用固定常量 `__SUMMARY__`）。

## 7. Webhook 异步回调（轻量替代 Kafka）

适合**没有 Kafka 集群 / 个人小团队 / 一次性集成**的场景。语义同 Kafka 的整批结束事件，但通道是 HTTP POST。

### 与 Kafka 通道的差异

| 项 | Kafka §6 | Webhook §7 |
| --- | --- | --- |
| 通道 | broker 持久化、消费方主动订阅 | HTTP POST，ai-phone 主动调被叫 |
| 事件粒度 | item 终态 + 批次收口（两类） | **仅批次收口**（每批 1 次） |
| 配置粒度 | 全局（一份 broker 配置） | **每批一个 URL**：投递时传 `callbackUrl` |
| 失败保证 | broker 持久 + at-least-once | **发一次就吞**（不重试） |
| 安全 | broker 鉴权 / SASL | 暂不签名（v1.8 简化版）；URL 自身保密即可 |
| 默认开关 | 后端配置启用 | 投递时不传 `callbackUrl` 就不发 |

> Kafka 与 Webhook **可以同时启用**——后端配了 Kafka，投递批次又传了 `callbackUrl`，则两路都发，互不影响。

### 触发条件

只要满足下面三条，本批 ai-phone 就会发一次 webhook：

1. 投递请求体里 `callbackUrl` 非空且合法（`http(s)://` 开头、长度 ≤ 1024）
2. 本批所有 item 都已落到终态（与 §6 `submission.terminal` 触发时机一致）
3. ai-phone 进程能向外建 TCP 连接（5s 内拨通）

### 推送 body（与 Kafka §6.4 一致）

ai-phone 会 POST 一份 `application/json; charset=utf-8`，body 即 §6.4 的 `submission.terminal` 完整 payload，字节级一致：

```json
{
  "event": "submission.terminal",
  "version": 1,
  "submissionId": "abcdef012345",
  "submissionName": "回归冒烟-2026-04-18 · 主流程",
  "submissionState": "done",
  "totalItems": 10,
  "counts":              {"success": 9, "failed": 1},
  "platformCounts":      {"android": 5, "ios": 5},
  "platformStateCounts": {"android": {"success": 4, "failed": 1}, "ios": {"success": 5}},
  "summaryReportUrl": "/files/reports/abcdef012345/_summary.html",
  ...
}
```

### 接收方实现要点

- **HTTP 响应必须返 2xx**，否则 ai-phone 这边记一条 WARN 日志，本批 webhook 当作丢失（不重试）
- **5s 内回包**，否则连接被拨断（与丢包等效）
- **接收处理要幂等**——虽然 v1.8 不重试，但接收方仍应按 `submissionId` 做去重，防止部署演进时重发
- **不验证签名**——v1.8 简化版没做 HMAC，URL 本身保密 + 只暴露内网即可。如需 HMAC，可在后续版本加 `Authorization` / `X-AiPhone-Signature` 头

### 想发 IM 群（飞书 / 钉钉 / 企微）怎么办

ai-phone 不内置各家 IM 协议适配。常见做法：在 `callbackUrl` 后挂一个**轻量转换 server**，把通用 JSON 翻译成各家卡片协议再转发。10 行左右 Node / Python 脚本即可，参考各家平台文档：

- 飞书 incoming webhook：`{"msg_type": "interactive", "card": {...}}`
- 钉钉自定义机器人：`{"msgtype": "markdown", "markdown": {...}}`
- 企微群机器人：`{"msgtype": "markdown", "markdown": {...}}`

### 接收方最小代码示例（Python / FastAPI）

```python
from fastapi import FastAPI, Request

app = FastAPI()

@app.post("/ai-phone-callback")
async def on_submission_done(req: Request):
    payload = await req.json()
    sub_id = payload["submissionId"]
    counts = payload["counts"]
    if counts.get("failed", 0) > 0:
        # 失败 case 多于 0 → 触发告警 / 发飞书 / 写日志
        ...
    return {"ok": True}        # 必须返 2xx
```

## 报告 URL 约定

每条 item 进入终态（`success` / `failed`）后，响应里的 `report_url` 就有值。

- 单 case 报告：`http://{server}/files/reports/{submissionId}/{itemId}__{platform}.html`
- 批次汇总报告（所有 item 终态后）：`http://{server}/files/reports/{submissionId}/_summary.html`

报告是**自包含 HTML**：

- 内联 CSS / JS，图片走服务端图床 URL
- 可以邮件转发、CI 归档、Jenkins / 飞书卡片直接嵌入
- 单 case 采用时间线视图（log 和 step 按时间戳归并）
- 批次汇总是 SPA：左侧用例列表，右侧复用单 case 模板，`caseName` 为主显示
- 点击任意截图可放大查看

## 测试清单（接入方建议跑一遍）

- [ ] **单平台单 case** → 能跑到 `success`，报告链接可打开，before / after 双图齐全

- [ ] **子集池**（`deviceAliasPools[p] = ["A1","B1"]`） → 哪台先 ready 哪台被消费；两台都跑起来时负载分担
- [ ] **多端拆条** → `platforms: ["android","ios","harmony"]` 展开后服务端落 3 条 SubmissionItem，响应 items 长度 = 3；`deviceAliasPools` 仅列 2 端时，未列的那端走池内任挑

- [ ] **过期查询** → 把 `submission_external_retention_days` 调小验证 `404 expired`
- [ ] **批次 summary 报告** → 左侧 case 列表 / 右侧时间线 / `caseName` 作为主显示
- [ ] **`rejectReason` 各分支** → 故意给错 `platforms` / 缺 `caseId` / `deviceAliasPools` 用错 key / 池含空串 对照报错

## 和内部 API 的区别

仅用于接入方心里有数，对外不暴露。

| 方面 | 内部 `/api/internal/submissions` | 外部 `/api/submissions`（本文档） |
| --- | --- | --- |
| 鉴权 | `Authorization: Bearer` | 匿名 |
| `origin` 字段 | `internal` | `external` |
| 15 天过期 | 不关心 | 过期后统一 `404 expired` |

两者共享同一个 `SubmissionScheduler` 实例和同一份 `Submission` / `SubmissionItem` 表，不会串扰。
