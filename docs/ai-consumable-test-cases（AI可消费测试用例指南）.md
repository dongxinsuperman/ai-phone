# AI 可消费测试用例指南

AI PHONE 可以直接执行一句自然语言目标，例如“打开设置并进入关于本机”。但在真实业务回归里，case 写法会显著影响执行稳定性。

如果你希望 AI PHONE 的 case 执行得更稳，建议把 `runContent` 写成 **AI 可消费测试用例**，而不是传统 QA 提纲。

完整方法论与公开 baseline skill 见：

[AI 可消费测试用例模式 / AI-Consumable Test Case Pattern](https://github.com/dongxinsuperman/ai-executable-case-pattern)

## 核心原则

传统 case 默认人类 QA 具备上下文；AI PHONE 执行时不应该依赖这些默认常识。

一条更适合 AI PHONE 的 case，应当只靠四个字段就能被执行到底：

- 测试标题：说明触发动作和核心验证点
- 前置条件：只放冷启动、登录归正、环境约束等执行基线
- 操作步骤：把入口路径、控件文案、输入值、动作顺序写清楚
- 预期结果：只保留一个核心视觉断言

## 推荐结构

```text
测试标题：【正向】手机号密码登录后首页右上角显示用户头像
前置条件：关闭 App「示例 App」（杀进程）后重新打开 App「示例 App」；必须按照步骤一步步执行
操作步骤：从启动页点击「登录」入口、在「手机号」输入框输入 13800138000、在「密码」输入框输入 Demo@123、点击「登录」按钮
预期结果：首页右上角显示用户头像
```

## 不推荐写法

```text
测试标题：【正向】用户登录
前置条件：用户已注册账号；位于登录页；网络正常
操作步骤：输入账号密码、点击登录按钮、若弹出协议勾选弹窗则勾选同意
预期结果：登录成功；跳转至首页；右上角显示用户头像；底部 Tab「我的」显示已登录态
```

这类 case 对人类 QA 可能够用，但对 AI PHONE 风险很高：

- “账号密码”没有具体值
- “位于登录页”没有入口路径
- “网络正常”没有可执行含义
- “若弹出则勾选”让执行器在运行时做业务决策
- 预期结果包含多个断言，失败时难以判断核心验证点

## 对 `runContent` 的建议

`runContent` 不要求固定格式；一句话、小段落、四字段 case 都可以投递。

但对于业务回归、兼容性测试、批量投递和轨迹缓存，建议采用四字段结构：

```json
{
  "caseId": "login_001",
  "caseName": "手机号密码登录后首页右上角显示用户头像",
  "platforms": ["android"],
  "runContent": "测试标题：【正向】手机号密码登录后首页右上角显示用户头像\n前置条件：关闭 App「示例 App」（杀进程）后重新打开 App「示例 App」；必须按照步骤一步步执行\n操作步骤：从启动页点击「登录」入口、在「手机号」输入框输入 13800138000、在「密码」输入框输入 Demo@123、点击「登录」按钮\n预期结果：首页右上角显示用户头像"
}
```

## 最小检查清单

投递复杂 case 前，先快速检查：

- 没有“测试账号”“某页面”“若干”“XXX”等占位词
- 没有“找不到则”“如果默认不对则”“重复以上规则”等业务分支
- 每个按钮、Tab、弹窗、卡片文案都有来源
- 异常 case 写明了异常注入工具和具体路径
- 预期结果只有一个核心断言
- 能从冷启动一路走到被测页面

## 和轨迹缓存的关系

轨迹缓存对起跑状态、账号状态、设备状态和页面稳定性要求更高。case 越像“可执行脚本”，缓存复跑越容易稳定。

如果 case 里包含模糊入口、临时判断、资源随机选择或多个断言，首跑即使成功，复跑也更容易漂移。

## 公司内部落地

公开 baseline 不应该原样套进所有公司。团队应结合自己的产品继续补齐：

- feature list / PRD 格式
- 测试账号与登录策略
- 测试数据构造工具
- 异常注入方式
- UI 证据来源
- 人工执行路由边界
