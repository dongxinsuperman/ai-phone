# 缓存回放步骤化日志改造方案

## 出发点

缓存回放本质上很简单：
**等页面稳定 → 截图 → 回放动作 → 判断成功还是失败 → (失败走分支) → 再截图 → 步骤结束**。

首跑读起来清楚，是因为每一拍都打一条短日志，标题就是环节。回放只要照同一种
节奏打，就能读懂。

之前的方案把这套节奏拧成了"步骤完成块"汇总——所有过程压进一条 9 行日志，
结果中间 12 秒空白看起来像卡，块本身和旧 RunStep 端点时间戳粘连导致顺序倒置。

这次回到最朴素的模型：**一拍一日志，每条单行**，跟首跑完全一致。

## 一个步骤的标准七拍

```text
#N 缓存步骤  — ━━ 第 N 步 / 共 M 步 ━━                  ← 第 0 拍：步骤端点（带 #N）
   缓存目标  — 点击同步刷题  ·  action_id=a001 type=click ← 第 0.5 拍：目标元信息（不带 #N）
   缓存稳定  — 等待页面稳定 ...                           ← 第 1 拍：等稳定
   缓存截图  — phase=before                              ← 第 2 拍：截图前
   缓存动作  — type=click intent=... point=...           ← 第 3 拍：派动作
   缓存执行  — 点击坐标 (x, y) 耗时=...ms                 ← 第 4 拍：执行
   缓存结果  — 成功 / 失败 reason=...                     ← 第 5 拍：判结果

[如果失败 → 走 V1/V2/V3 各自的分支，仍然每一拍一条单行]

   缓存截图  — phase=after                              ← 第 6 拍：截图后
#N 缓存完成  — ━━ 第 N 步 完成 ━━  elapsed=...ms status=... ← 第 7 拍：步骤端点（带 #N）
```

每条日志的硬性要求：

- **单行 content**，不允许多行汇总。
- **标题就是环节名**，不要在 content 里再写 `校验：...`、`动作：...` 这种
  "字段标签" 风格——标题本身已经说明了，content 直接放事实。
- 步骤端点（`缓存步骤` / `缓存完成`）必须带 `━━` 包裹，跟首跑视觉一致。

### `#N` 前缀约定（2026-05-16 二轮改造）

为了让缓存日志和首跑一样可以**眼睛快速锚定第几步**，端点行必须带 `step=index`
字段；前端拿到 step 字段才会渲染 `#N` 前缀。

| 行类型 | 是否带 step | 视觉效果 |
|---|---|---|
| `缓存步骤` / `缓存完成`（端点） | **必须带** `step=index` | `#1 缓存步骤 — ━━ 第 1 步 / 共 3 步 ━━` |
| `缓存目标` / `缓存稳定` / `缓存动作` / `缓存执行` / `缓存截图` / `缓存路标` / `缓存定位` / `缓存辅助` / `缓存修复` / `缓存结果`（过程） | **不带** | 仅显示标题 + content，无前缀 |
| `缓存回放` / `缓存断言`（全局） | **不带** | 同上 |

落地点：`service.py:_log(level, title, content, step=None)` —
`ReplayRunner._log` / `V3ReplayRunner._log` 都接 step 关键字参，最终塞进
`log_event(..., step=step)`。

### `缓存步骤` content 瘦身约定（2026-05-16 二轮改造）

`缓存步骤` content 只能是 `━━ 第 N 步 / 共 M 步 ━━`，不允许再回退到
旧版 `━━ 开始第 N 步 / 共 M 步 ━━ 目标=... action_id=...` 那种多塞元
信息的格式——目标 / action_id / type 必须单独成行 `缓存目标`。

理由：
1. 首跑的 `━━ 第 N 步 ━━` 是纯标题，扫日志时眼睛锚点最强；缓存对齐这套节奏。
2. 目标信息独立成行不丢任何排查能力，反而比"标题里堆 3 段"更易读。
3. 端点 + 目标两行加在一起仍然只占 2 行，不影响"步骤化"密度（七拍变七拍半）。

## 步骤之间的边界

为了让读日志的人不会把上一步的尾巴和下一步的头混淆，必须遵守这个顺序：

```text
... 上一步流程 ...
   缓存截图  — phase=after                  ← 上一步第 6 拍
#1 缓存完成  — ━━ 第 1 步 完成 ━━ ...        ← 上一步第 7 拍（端点带 #N）
#1 第 1 步完成 · click — ...                 ← 旧 RunStep 端点（保留）
#2 缓存步骤  — ━━ 第 2 步 / 共 3 步 ━━       ← 下一步第 0 拍（端点带 #N）
   缓存目标  — 进入第一个习题卡片  ·  ...    ← 下一步第 0.5 拍
   缓存稳定  — ...
```

`缓存完成` 必须在 `#N 截图 phase=after` **之后**触发，在 `_emit_step_end()`
（会打 `#N 第 N 步完成`）**之前** 触发。这条顺序由代码触发位点保证，不靠
时间戳。

## V1 / V2 / V3 的差异只在"失败分支"

三个版本的成功路径**完全一样**（都是上面那七拍）。区别只在第 5 拍判结果
为失败时怎么处理：

### V1 失败：直接失败

V1 是固定动作，失败就失败，不重试。

```text
缓存结果    — 失败 reason=...
缓存完成    — ━━ 第 1 步 完成 ━━  elapsed=...ms status=失败
```

### V2 失败：路标 → 真实间隔 → 辅助 VLM

V2 在执行后还有一层路标比对，失败时多走几拍：

```text
缓存路标    — 执行后截图比对 action_id=a002, 首次真实间隔=29429ms
缓存路标    — 不一致 global=0.12 center=0.02 reason=global>0.03

缓存稳定    — 缓存路标不一致，按首次真实间隔再等待 5493ms
缓存路标    — 等待后再比对，对齐成功 global=0.00

[如果仍不一致 → 辅助 VLM]

缓存辅助    — 请求 recovery VLM
缓存辅助    — recovery 判=WAIT_MORE 等待 5000ms
缓存辅助    — recovery 判=REPAIR_ACTION 修复动作=click point=(540,1480)
缓存修复    — type=click point=(540,1480) intent=关闭弹窗
缓存路标    — 修复后再比对，对齐成功
```

成功汇合到第 6 拍 `缓存截图 phase=after`。

V2 的瞬态 gate（可跳过动作）：

```text
#3 缓存步骤  — ━━ 第 3 步 / 共 5 步 ━━
   缓存目标  — 关闭启动弹窗  ·  role=optional_ephemeral
   缓存辅助  — 可跳过瞬态动作，请求 ephemeral gate
   缓存辅助  — gate 判=SKIP，当前页面已可衔接下一步
#3 缓存完成  — ━━ 第 3 步 完成 ━━  elapsed=2402ms status=已跳过(瞬态)
```

### V3 失败：rescue VLM

V3 是定位失败时多走几拍：

```text
缓存定位    — 正在当前截图中寻找「点击 Futures 标签」
缓存定位    — 未找到 reason=v3 locator 返回屏幕边缘坐标

缓存辅助    — 请求 rescue VLM
缓存辅助    — rescue 判=WAIT 等待 3000ms
缓存定位    — 等待后重新寻找，已找到 坐标=(538, 2244)
```

rescue 关闭遮挡：

```text
缓存辅助    — rescue 判=POPUP_CLOSE 修复动作=click point=(540,1480)
缓存修复    — type=click point=(540,1480) intent=关闭弹窗
缓存定位    — 修复后重新寻找，已找到 坐标=(...)
```

rescue 放行（当前页面已可衔接）：

```text
   缓存辅助  — rescue 判=CONTINUE_REPLAY 当前页面已可衔接下一步
#2 缓存完成  — ━━ 第 2 步 完成 ━━  elapsed=... status=已跳过(无需操作)
```

## 标题清单

只有这些标题，没有其他：

| 标题 | 用途 | 带 #N 前缀 |
|---|---|---|
| `缓存回放` | 整段回放开始 / 结束（外层） | 否 |
| `缓存步骤` | 第 N 步开始（端点） | **是** |
| `缓存目标` | 目标 + action_id + type（拆自 `缓存步骤`） | 否 |
| `缓存稳定` | 等待页面稳定 | 否 |
| `缓存截图` | before / after 帧 | 否 |
| `缓存动作` | 派发缓存动作 | 否 |
| `缓存执行` | 实际执行（点击 / 输入 / 滑动 / 等待） | 否 |
| `缓存结果` | 本拍动作判定（成功 / 失败） | 否 |
| `缓存路标` | V2 路标比对（仅 V2） | 否 |
| `缓存定位` | V3 重新定位（仅 V3） | 否 |
| `缓存辅助` | 辅助 VLM 介入（recovery / rescue / ephemeral） | 否 |
| `缓存修复` | 辅助给出的修复动作执行 | 否 |
| `缓存完成` | 第 N 步结束（端点） | **是** |
| `缓存断言` | 全部步骤后的最终断言 | 否 |

旧标题继续保留落库（兼容已有测试和排查习惯）：`轨迹缓存` / `轨迹缓存 action` /
`轨迹缓存状态路标` / `轨迹缓存 VLM 介入` / `轨迹缓存修复动作` /
`轨迹缓存瞬态动作` / `轨迹缓存观察延迟` / `V3寻找目标` / `V3执行缓存动作` /
`V3局部辅助判断` / `V3局部辅助动作` / `V3页面稳定检测`。

但**主线只看新标题**——读日志只扫 `缓存步骤` 到 `缓存完成` 之间的新标题
就能完整还原一个动作的故事。旧标题作为底层证据，不再承担"用户阅读主线"。

## 实施约束

1. V1 / V2 只改 `backend/ai_phone/server/trajectory_cache/replay.py`；
   V3 只改 `backend/ai_phone/server/trajectory_cache/v3_replay.py`。
2. **必须把当前的 `缓存步骤完成` 多行汇总块退化为单行**，只保留
   `elapsed` 和 `status`。
3. **必须流式输出每一拍**，禁止把多个时间点合并成一条多行日志。
4. **必须保留所有旧日志的实时输出**（不要为了"步骤化"把它们静音或合并）。
5. **必须遵守步骤间顺序**：`缓存完成` 在 `#N 截图 phase=after` 之后、
   `_emit_step_end()` 之前触发。
6. 不动 driver action、坐标转换、路标比对、辅助模型决策、保存阶段、首跑日志。
7. 不抽跨版本公共执行器，最多抽纯文本格式化小函数。
8. **端点行（`缓存步骤` / `缓存完成`）必须通过 `step=index` 传给 `_log`**，
   其他过程行不传——见上表"`#N` 前缀约定"。
9. **`缓存步骤` content 只能是 `━━ 第 N 步 / 共 M 步 ━━`**，不允许塞
   目标 / action_id / type，那些必须独立成行 `缓存目标`。

## 验收

读一段回放日志，**只看新标题（`缓存*`）那一列**，应该能像下面这样
完整还原一个动作，并且端点行能一眼看到 `#N`：

```text
#1 缓存步骤  开始（端点带 #N）
   缓存目标  点击同步刷题  ·  action_id=a001 type=click
   缓存稳定  等稳定
   缓存截图  before
   缓存动作  派 click
   缓存执行  点击坐标
   缓存结果  成功
   缓存截图  after
#1 缓存完成  结束（端点带 #N）
```

如果其中任何一拍**缺失**、**和下一步混在一起**、**被压缩进了一条
多行日志**，或者**端点行没带 `#N`**，就是改造失败。
