## 2. 问题 1：瞬态 UI 是否影响正常 step 循环时间

### 2.1 结论

会影响，而且是当前 step 主路径上的同步耗时，不是后台异步行为。

但要区分两种阶段：

1. 普通 click 后做“瞬态 UI 检测”
2. 下一步命中 takeover 后做“复用缓存帧 + 重唤起接管”

### 2.2 影响点拆分

#### A. 普通 click 后的检测会拖慢当前 step

在普通非链式 click / long_press / double_tap 后，runner 会在 step 尾部调用 `detect_transient_ui(...)`。

这段逻辑会：

- 先用当前 `tail_bytes` 当 early 帧
- 再额外 sleep `TRANSIENT_LATE_DELAY_MS`
- 再抓一张 late 帧
- 再做三段式 pHash 判定

当前默认值：

- `TRANSIENT_LATE_DELAY_MS = 3000`
- 注释里也明确写了副作用：“每次 click 类动作后单步增加 ~3s 等待”

这意味着：

- 不是只有“真的命中瞬态 UI”才增时
- 而是所有进入这条检测分支的普通 click 类 step，都会多一段同步等待

### 2.3 哪些 step 会触发这段额外耗时

只有满足以下条件时才进入检测：

- 当前 step 不是 takeover 步
- 当前 step 不是链式动作
- 当前动作是 `click / long_press / double_tap`
- 当前动作执行后成功拿到了 `tail_bytes`

所以：

- 普通点击页签、按钮、列表项，理论上都会被这段检测扫一遍
- `scroll / drag / type / wait / finished / assert_fail` 不走这段
- 瞬态 UI 接管步本身也不会再次触发，避免套娃

### 2.4 takeover 步的时间表现

当上一步已经检测到瞬态 UI 后，下一步会进入 takeover：

- 不再做稳定检测
- 不抓新 before 帧
- 直接复用缓存的 `visible_frame` 给 VLM
- 执行时先做一次“重唤起 click”
- 再等待 `TRANSIENT_TAKEOVER_WAIT_MS = 500ms`
- 再立即执行 VLM 的目标 click

所以 takeover 步：

- 省掉了“稳定检测 + 抓新 before 帧”的时间
- 但增加了一次系统级 click + 500ms wait

整体上看：

- “普通 click 检测步”是在当前 step 末尾付出约 3s 成本
- “下一步 takeover 步”是在下一步开头省掉稳定检测，但在执行前增加 1 次重唤起和 500ms 等待

### 2.5 对整体循环的实际影响判断

我的判断是：

- 瞬态 UI 机制确实会拉长很多普通 click step 的耗时
- 它不是只在视频场景触发的“极小局部补丁”，而是 click 类动作后的一段同步观察成本
- 但在“视频工具栏这类必掉控件”的场景下，这个成本是值得的，因为没有它下一步几乎必定点空

### 2.6 后续可优化方向

短期不急改，但可以记住这几个方向：

1. 给瞬态检测加更严格的前置门槛
只对“可能是视频/播放器/相机/直播页”的场景触发，而不是对所有 click 类动作统一扫

2. 把 `TRANSIENT_LATE_DELAY_MS` 配置化分平台/分 App
不同播放器寿命不同，固定 3000ms 会让很多场景白白多等

3. 未来考虑异步检测
当前注释里也写了，难点是抓帧与主循环资源竞争；这不是现在最优先，但方向是对的

### 2.7 当前决定落地的收敛方案

为了不让瞬态 UI 机制继续全量拖慢普通 case，本轮决定先落一个保守实现：

1. 加 env 总开关

- `AI_PHONE_TRANSIENT_UI_ENABLED=true|false`
- `false` 时，本次 run 无论输入内容是什么，都完全关闭瞬态 UI 检测/接管

2. 加“动态判断系统”

- 不是模型，不走审判逻辑
- 只在 run 启动时做一次本地字符串命中判断
- 命中白名单关键词时，才给本次 run 挂上“允许启用瞬态 UI”的标记

3. 白名单关键词先放 env

- `AI_PHONE_TRANSIENT_UI_KEYWORDS=视频,播放,播放器,...`
- 这样后续可以按具体 case 增量补词，而不用先设计复杂配置中心

4. 判定时机借鉴“起跑线”实现

- 都是在 run 开始时先做一次系统侧预处理/预判
- 不是每个 step 都重新判断，避免行为抖动

这样做的目标是：

- 普通 case 保持快
- 视频/播放器类 case 保留瞬态能力
- 关键词可以马后炮式增量补齐，先保守后扩充

- env 开启但 goal 未命中白名单时，也不启用瞬态 UI

### 2.9 关于“多动作链”的最新修正

这轮又确认了一点：

- `多动作链` 与 `动态系统 / 瞬态 UI 接管` 没有前置依赖关系
- 多动作链的真实触发语义非常简单：
  - 主 VLM 在同一轮里输出了多条 `Action:`
  - runner 就按链处理
- 它不是“只有动态系统介入时才会触发”的能力

因此，主提示词里原先那种“链式动作主要给瞬态 UI 用”的表述会误导主 VLM。

本轮已把提示词改成：

- `链式动作` 是一个通用的“两连击能力”
- 既可以用于瞬态 UI
- 也可以用于“点击【复习】后马上点击【加餐】”这类同屏顺序点击
- 只要满足：
  - 第 1 击后第 2 击坐标仍然有效
  - 中间不需要重新看图决策

这样做的目的不是让 VLM 滥用链，而是把它从“视频特化技巧”纠正为“通用同步两连击工具”。

### 2.10 本轮补充：链式动作提示系统（已回滚，不作为当前方案）

曾尝试过一个更激进的方向：

- 在 run 启动时扫描 goal
- 若命中“马上 / 立即 / 双动作链 / 连续点击”等词
- 就向主 VLM 注入一次“优先输出双 Action”的运行时提示

但这条路最后没有保留，原因是：

- 侵入性偏大
- 对主 VLM 的行为偏置过强
- 不符合“先尽量靠 prompt 语义约束，而不是再叠本地运行时强提示”的当前边界

所以当前正式保留的方案仍然是：

- 只在主提示词里把链式动作表述成通用两连击能力
- 不再额外增加 run 级链式动作提示系统

## 3. 问题 2：增加 `finished()` 的断言系统

### 3.1 结论

我认为非常有必要，而且优先级高。

当前系统的真实现状是：

- 结构化通道、审判系统、卡死检测，主要都在保护“过程”
- 但 `finished()` 最终是否真的满足断言，没有独立的系统级结果验收
- 主 VLM 一旦说 `finished(...)`，runner 直接按成功结束

这就导致：

- VLM 如果在最后一步“自我说服”
- 即便 thought 里已经出现自相矛盾
- 只要它最终吐了 `finished(...)`
- 结果仍然会落成成功

本次 case 就是这个问题的典型表现。

### 3.2 断言系统的目标

在主 VLM 输出 `finished(...)` 时，再走一层“独立、只读、不可执行动作”的验收模型。

断言系统只做一件事：

- 判断“当前截图是否足以支持 finished 的成立”

它不负责继续操作，不负责规划路径，不负责修复流程，只负责裁决：

- PASS
- FAIL
- UNSURE

### 3.3 我建议的接入时机

接入在 runner 主循环里，`ACTION_FINISHED` 分支返回成功之前。

也就是现在这段逻辑：

- 先拿到 `finish_msg`
- 不是立刻 `return True, finished: ...`
- 而是先调用 `_verify_finished(...)`
- 根据判官结果决定：
  - PASS：维持成功
  - FAIL：改写为失败
  - UNSURE：不给过，回灌提示给主 VLM 继续看图或转 `assert_fail`

### 3.4 断言系统输入建议

建议最小输入包含：

1. 原始 goal
2. 若是结构化通道，则提取后的“预期结果清单”
3. 主 VLM 本步的 `Thought`
4. 主 VLM 本步的 `finished(content=...)`
5. 当前最终截图（就是现在 `finish_ok` 前那张图）
6. 最近 3-5 步动作摘要

可选增强输入：

7. 当前 step 的 before 图
8. 目标区域裁剪图
9. 若未来接 OCR，可附 OCR 文本

### 3.5 判官输出协议建议

建议强约束成非常窄的协议：

```text
PASS: 原因
FAIL: 原因
UNSURE: 原因
```

其中：

- `PASS`：截图能直接证明预期已满足
- `FAIL`：截图能直接证明预期未满足，或关键条件缺失
- `UNSURE`：截图证据不足，不能支持 finished

注意：

- `UNSURE` 不应该当成功
- 对结构化 case，`UNSURE` 更应当偏保守

### 3.6 结果改写策略

我建议：

#### 结构化 case

- `PASS` -> 保持 `finished`
- `FAIL` -> 直接改写成 `assert_fail`
- `UNSURE` -> 不允许成功，回灌提示给主 VLM 再做 1-2 步确认；若仍 UNSURE，再失败

理由：

- 结构化 case 本来就要求“每条预期结果必须能从截图直接验证”
- 那么不能证明就不能判成功

#### 自由对话 case

可以更宽一点：

- `PASS` -> 成功
- `FAIL` -> 失败
- `UNSURE` -> 允许再补一步确认，避免太严导致体验恶化

### 3.7 本次 case 下断言系统会怎么裁

如果把本次最终图 + 原始要求 + `finished(content=...)` 一起交给判官，我认为更大概率会给：

```text
FAIL: 目标视频卡片进度条未达到约 30%，且百分比证据不足，finished 说明与截图不一致
```

也就是说，这一层确实能拦住这次误判。

### 3.8 断言系统的价值

它最重要的价值不是“模型更聪明”，而是“把 finished 的解释权从主 VLM 手里收回来一部分”。

主 VLM 擅长：

- 导航
- 找入口
- 执行动作

但它在最后一步很容易：

- 补全叙事
- 自我合理化
- 用模糊语言把不满足说成“差不多满足”

断言系统就是专门兜这个洞。

### 3.9 落地建议（分阶段）

#### Phase 1：最小可用版本

只在结构化通道下启用 finished 判官。

实现最小闭环：

- `_verify_finished(...)`
- 只读模型调用
- 三态协议 PASS/FAIL/UNSURE
- FAIL 改写结果

#### Phase 2：补强日志与报告

在 run log / HTML 报告里补：

- `断言校验 · PASS`
- `断言校验 · FAIL`
- `断言校验 · UNSURE`

并记录：

- 主 VLM 的 finished 文案
- 判官理由
- 最终结果是否被改写

#### Phase 3：细化到结构化断言条目

对结构化 case，把“预期结果”拆成多条逐项验收。

例如：

- 条件 1：已回到章节列表页
- 条件 2：左上角显示进行中标识
- 条件 3：视频进度条约 30%
- 条件 4：做题进度仍为 0/N

断言系统返回：

- pass_items
- fail_items
- unsure_items

这样会比一句总判更适合调试。

### 3.10 本轮已实际完成的最小落地版本

结合“系统较脆、不要影响既有流程”的约束，本轮断言系统先做成了一个保守版本。

1. 触发时机

- 只在主 VLM 输出 `finished()` 后触发
- 不允许断言系统继续执行任何 step
- 不让主 VLM再补动作

2. 当前流程

- 主 VLM 正常把整个流程跑完
- 当它申请 `finished()` 时，runner 先记：
  - `主VLM申请完成`
- 然后进入：
  - `断言系统 · 复核`

3. 当前裁决协议

- 目前只接收：
  - `PASS: ...`
  - `FAIL: ...`

4. 当前结果处理

- `PASS`
  - 维持成功
  - 记录 `断言系统 · 通过`
- `FAIL`
  - 直接把最终结果改写为 `assert_fail`
  - 记录 `断言系统 · 驳回`

5. 当前保守兜底

- 如果断言系统调用失败 / 配置缺失 / 协议异常：
  - 先不打死 run
  - 打清晰日志：
    - `断言系统 · 调用失败`
    - `断言系统 · 跳过`
    - `断言系统 · 协议异常`
  - 然后回退采纳主 VLM 的 `finished`

这样做是为了先把“finished 不再直接拥有最终解释权”这条主链接通，同时避免因为断言系统自身的基础设施问题，把现有流程整体误伤。

6. 已补测试

- `finished()` 被断言系统驳回时，最终结果应改写为 `assert_fail`
- `finished()` 被断言系统通过时，最终结果应保持成功

### 3.11 关于“断言系统”的最新修正

这轮通过“点击【复习】后马上点击【加餐】”这个 case，又暴露出一个新问题：

- 断言系统之前过度偏向“单张最终截图必须能证明全部历史过程”
- 这会把“过程不可回溯”误判成“结果未达成”

但断言系统的职责，应该是：

- 只裁决**最终结果**
- 不负责复盘历史顺序
- 过程偏航由审判系统负责

因此本轮已把断言系统提示词修正为：

1. 以最终截图里的最终结果为最高优先级
2. `goal` 与 `finished` 文本只用于帮助理解“应该验什么结果”
3. 如果是“先点 A 再点 B”“点击 X 后进入 Y”这类过程描述：
   - 只要最终截图已经清楚显示结果成立，就应该 `PASS`
   - 不得仅因“单张截图无法证明点击顺序”而 `FAIL`
4. 只有当最终截图和目标结果明显矛盾时，才 `FAIL`

这次改动的核心思想是：

- `审判系统` 管过程
- `断言系统` 管结果

两者职责边界分开，避免断言系统越权去做“历史过程审查”。

### 3.12 本轮补充：断言系统结果铁律（本地硬拦截方案已回滚）

后续又出现了两个自由目标：

- “点击加餐后马上点击预习”
- “双动作链：点击加餐后马上点击预习”

这类目标的共同点是：

- 最终截图可以证明最终结果（例如当前选中的是“预习”）
- 但单张截图无法回放“是否真的先点过加餐”

如果断言系统仍然因此 FAIL，就说明它还在越权做“过程审查”。

因此曾尝试过一个本地硬拦截方向：

- 若断言系统给出 `FAIL`
- 且理由本质上只是“无法证明历史过程 / 顺序 / 是否用了双动作链”
- 就在本地直接忽略这次驳回

但这条路后来也回滚了，因为你明确要求：

- 不要再在本地代码里替断言系统二次改判
- 应该直接把 prompt 写成铁律
- 让断言系统自己只做“结果断言”，而不是靠外层补丁兜

所以当前正式保留的方向是：

- 不做本地过程型驳回忽略器
- 只加强断言系统 prompt 的硬约束力度
- 让它从源头上避免再产出这类越权驳回

### 3.13 当前最终定版：只校验“最后一个 action”的结果

在继续观察真实 run 后，进一步收紧为一个更直接的铁律：

- 断言系统的有效输入只看：
  - 用户原始目标
  - 最后一张截图
- 裁决时只判断：
  - 用户目标里的**最后一个 action 执行结果**
  - 是否已经在最后截图中成立

例如：

- `点击加餐后马上点击预习`
  - 只看最后一个 action：`点击预习`
  - 只验证截图是否已经体现“预习已被选中/进入预习结果态”

- `点击复习后马上点击加餐`
  - 只看最后一个 action：`点击加餐`
  - 只验证截图是否已经体现“加餐已被选中/进入加餐结果态`

这意味着：

- 前面的动作、顺序、中间过程、是否真的点过某个过渡按钮
- 都不再属于断言系统的审查范围

断言系统现在的职责被进一步压缩成：

- **只验最后一步结果**
- **不验历史过程**

### 3.14 当前最新定版：结构化 / 自由对话两套断言口径

继续跑真实 case 后，确认“一刀切只验最后一个 action”并不适合结构化测试用例。

因此当前最终定版改为分流：

1. 结构化 case

- 断言系统回到最初的严格模式
- 只根据最终截图，逐条验证“预期结果”
- 看不出来就是 `FAIL`
- 不允许脑补，不允许“可认为仍为”“应该还是”
- 任一关键预期结果证据不足，都必须 `FAIL`
- 但这里的“严格”不是逐字匹配，而是：
  - 只围绕“预期结果”本身验收
  - 允许同义表达、界面别名、常见产品话术变体
  - 语义等价可以通过，证据不足不能通过

2. 自由对话

- 继续使用“只验最后一个 action 结果”的宽松口径
- 不复盘历史过程
- 不审查点击顺序

也就是说，现在的边界是：

- `结构化通道`：严格截图验收
- `自由对话通道`：最后一步结果验收

## 4. 问题 3：真实等待改成实时不急

### 4.1 结论

这个问题我同意你现在不急着改真实行为，但必须先把“裁剪事实”暴露出来。

本次 case 的关键偏差之一是：

- 主 VLM 说 `wait(seconds=245)`
- 系统实际执行的是 60 秒
- 但后续模型仍沿用“我已经等到 245 秒附近”的叙事

这会直接污染后续断言。

### 4.2 我建议的低成本修正

即便先不改真实等待行为，也建议先补两件小事：

1. 日志明确写“原请求 vs 实际执行”

现在只写：

### 4.3 本轮已实际完成的调整

这轮已经把 wait 改成“活的等待”，但仍保留硬上限：

- 上限从原来的 `60` 秒提升到 `1800` 秒
- 也就是最多等待 `30 分钟`

同时补了日志可解释性：

- 如果模型请求值没有超上限：
  - 记录 `等待 N 秒（模型指定/Thought兜底/默认轮询）`
- 如果模型请求值超过上限：
  - 记录 `请求等待 X 秒，实际执行 1800 秒（...；上限 1800 秒）`

这样后续就不会再出现：

- 模型以为自己等了很久
- 系统实际只等了很短
- 但日志又看不出被裁剪

本轮还补了一条测试，确保：

- 超过 30 分钟的 wait 会被裁到 `1800`
- 同时保留原始请求秒数，供日志展示

- `等待 60 秒（模型指定）`

建议改成：

- `等待 60 秒（模型指定 245 秒，受 MAX_WAIT_SECONDS=60 限制被裁剪）`

2. 把裁剪事实回灌给主 VLM

例如在 `pending_hints` 里加一条：

- `⚠️ 你请求等待 245 秒，但系统本次最多只执行了 60 秒。请基于实际等待结果重新判断，不要假设已经走完 245 秒。`

这样即便暂时不改真实等待，也能显著降低“时间错觉导致 finished 误判”的概率。

### 4.3 为什么这次没有正常抛失败

我认为主要是两层原因叠加：

1. 实际等待时间不足，但模型没有被强提醒
2. 最终没有 finished 断言系统，主 VLM 自我说服后直接成功收口

如果只修一个，我优先修第 2 点。

## 5. 建议优先级

我建议按下面顺序推进：

1. 先做 `finished()` 断言系统
这是当前“误报成功”的最大风险点，收益最高

2. 再补 wait 裁剪可视化与回灌提示
这是本次 case 的重要诱因，修起来也轻

3. 最后再考虑瞬态 UI 的时延优化
它重要，但不如前两项直接影响结果真伪

## 6. 最终判断

### 关于问题 1

瞬态 UI 会影响正常 step 的循环时间，而且影响是同步的、真实的；普通 click 后的检测会额外增加约 3 秒。

### 关于问题 2

强烈建议做 finished 断言系统，而且建议由它拥有“推翻主 VLM finished 结果”的权力。

### 关于问题 3

真实等待先不改没问题，但应尽快补上：

- 裁剪日志
- 回灌提示

否则类似本次这种“主 VLM 以为等够了，其实没等够”的误判还会继续出现。
