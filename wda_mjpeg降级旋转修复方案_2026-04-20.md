## 1. 问题陈述

`wda_mjpeg` 后端下物理旋转 iPhone 时，Web 端画面**方向不跟随、出现挤压/拉伸**。只有用户**退出工作台再进入**（或 `F5` 刷新）才能恢复正常。

原因链：

1. WDA 的 MJPEG `FBMjpegServer` 在单次 `xcodebuild test` 生命周期内，画布（canvas）尺寸取决于**首次建立 screenshot broadcast session 时**的设备方向；
2. 设备物理旋转后，WDA 的 `XCUIScreen.mainScreen().screenshot()` 已经返回新方向的截图，但**画布像素尺寸并未改变**——竖屏内容被硬塞进横屏画布（或反之），多余区域补黑，有效像素被挤压；
3. JPEG 单帧尺寸没变 → agent 端 `_last_size` 检测不到变化 → 不触发 `FMp4Streamer.restart()` → 前端 MSE init segment 的 `width/height` 也不变 → 前端 `useMseMirror` 的 `dimsChanged=false` → `MediaSource` 不重建，画面持续变形。

关键观察：**agent 单纯断开 HTTP 再重连**（现有 `_orientation_watch_loop` + `_need_reconnect` 机制）并**不足以**让 WDA 重建 canvas；必须要有 settings 重发 + usbmux 新连接 + 前端 `MediaSource` 重建三者同步才能修复。

---

## 2. 方案总览

**核心思路**：在 agent 检测到 WDA orientation 变化时，**整条链路推倒重建**，模拟"浏览器退出工作台再进入"的全部动作。

对 run 执行的影响：**零**——mirror streamer 是独立线程，与 VLM run 使用的 `driver.click()` / `driver.screenshot()` 走不同的 `WdaClient` 实例，互不干扰；重建过程约 1–2 秒画面黑屏，`log` / `step_done` / WebSocket 连接都不中断。

对前端 DOM 状态的影响：只销毁并重建 `MediaSource`，不重挂 `<video>` 元素，不重连 WebSocket，不丢失日志面板状态。

## 3. 详细设计

### 3.1 新增协议消息 `MSG_VIDEO_RESET`

`shared/protocol.py`：

```python
# MSE 链路强制 reset 信号：agent 端把 mjpeg/ffmpeg/usbmux 一条链推倒重建时
# 提前通知浏览器拆掉 ``MediaSource``，等待下一条 ``video_init`` 重建。
MSG_VIDEO_RESET = "video_reset"


class VideoResetMsg(TypedDict, total=False):
    type: Literal["video_reset"]
    serial: str
    reason: str  # 'rotate' / ...
    ts: float
```

记得把 `VideoResetMsg` 加进 `AgentToServer` 联合类型。

### 3.4 `IosMjpegStreamer` 新增 `_full_rebuild`

`agent/mirror/ios_capture_mjpeg.py`：

**构造函数**新增两个参数：

```python
def __init__(
    self,
    ...,
    on_reset: Optional[Callable[[str], None]] = None,
    rebuild_on_rotate: bool = True,
) -> None:
    ...
    self._on_reset = on_reset
    self._rebuild_on_rotate = bool(rebuild_on_rotate)
    self._need_reconnect = threading.Event()
    self._need_rebuild = threading.Event()  # 新增
```

**`_pump_loop`** 顶部加 rebuild 检查，URL 挪进循环（因为端口会变）：

```python
while not self._stopped:
    if self._need_rebuild.is_set():
        self._need_rebuild.clear()
        self._need_reconnect.clear()
        self._full_rebuild(reason="rotate")
        settings_pushed = False
    url = f"http://127.0.0.1:{self._mjpeg_local_port}/"
    ...
```

`iter_bytes` 内的中断条件改成同时检查两个 Event：

```python
if self._need_rebuild.is_set() or self._need_reconnect.is_set():
    ...
    self._need_reconnect.clear()
    break
```

**`_orientation_watch_loop`** 按开关选 flag：

```python
elif orient and orient != self._last_orientation:
    self._last_orientation = orient
    if self._rebuild_on_rotate:
        self._need_rebuild.set()
        self._need_reconnect.set()  # 同时让 iter_bytes 立刻跳出
    else:
        self._need_reconnect.set()  # 旧行为
```

**新增 `_full_rebuild` 方法**：

```python
def _full_rebuild(self, *, reason: str = "rotate") -> None:
    """模拟"浏览器退出工作台再进"的全部动作：
    1) 前置通知浏览器拆 MediaSource（避免新 init 到达前继续 decode 旧流）
    2) 停老 usbmux forwarder，新建一个（新本地端口 → WDA 看见新 client）
    3) 重新 push appium settings（触发 WDA 重建 mjpeg broadcast session）
    4) FMp4Streamer.restart() 清空 ffmpeg 编码状态
    5) _last_size 清零；pump_loop 下一轮用新端口重新 GET
    """
    if self._stopped:
        return
    logger.info("[{}] 开始整条链路重建 reason={}", self._log_tag, reason)

    if self._on_reset is not None:
        try:
            self._on_reset(reason)
        except Exception as exc:
            logger.debug("[{}] on_reset 回调异常: {}", self._log_tag, exc)

    if self._forwarder is not None:
        try:
            self._forwarder.stop()
        except Exception:
            pass
        self._forwarder = None
    try:
        self._setup_port_forward()
    except Exception as exc:
        logger.warning("[{}] rebuild 端口转发启动失败: {}", self._log_tag, exc)
        return

    try:
        self._push_wda_mjpeg_settings()
    except Exception as exc:
        logger.debug("[{}] rebuild push settings 失败（非致命）: {}", self._log_tag, exc)

    try:
        self._fmp4.restart()
    except Exception as exc:
        logger.warning("[{}] rebuild fmp4 restart 失败: {}", self._log_tag, exc)

    self._last_size = None
    logger.info("[{}] 重建完成，等待新一轮 mjpeg 帧到达", self._log_tag)
```

### 3.5 `_IosMirrorSession` 新增 `_on_mirror_reset`

`agent/main.py`：

```python
def _on_mirror_reset(self, reason: str) -> None:
    if self._stopped:
        return
    payload = {
        "type": P.MSG_VIDEO_RESET,
        "serial": self.serial,
        "reason": reason,
        "ts": time.time(),
    }
    logger.info("iOS mirror 请求前端 MSE 重建 serial={} reason={}", self.serial, reason)
    self._dispatch(payload)
```

### 3.6 Builder 传 `on_reset` 和 `rebuild_on_rotate`

`agent/mirror/__init__.py::build_ios_streamer` 的 `wda_mjpeg` 分支：

```python
return IosMjpegStreamer(
    ...,
    on_reset=on_reset,  # 新增参数
    rebuild_on_rotate=bool(s.wda_mjpeg_rebuild_on_rotate),
)
```

`passthrough` 分支不需要（每帧独立）；若父类 `IosMjpegStreamer` 的 `on_reset` 默认值设为 `None`，passthrough 直接继承即可，orientation watcher 不会对它造成副作用（因为 passthrough 完全 override 了 `_consume_jpeg` 不依赖尺寸比较链）。

`_IosMirrorSession.start()` 里调 builder 时把 `self._on_mirror_reset` 传过去。

## 4. 风险与不确定性

1. **`_push_wda_mjpeg_settings` 不一定能让 WDA 重建 canvas**  
   Appium WDA 的 `FBMjpegServer` 源码里，settings 变化会触发 `screenshotsBroadcaster` 重建。但不同 WDA fork / 版本行为可能不同。实施时要先看 `FBMjpegServer.m` 和 `FBSettings.m`，必要时改成"主动 stop + 重启 WDA session"的更重方案。

2. **重建过程中黑屏时长**  
   约 1–2 秒（usbmux 新连接 + WDA 建 session + ffmpeg 新 GOP），视觉体验比 passthrough 差一截。用户能接受就行，不能接受就别上这条路。

3. **多次快速旋转**  
   用户连续转几次手机可能触发多次 rebuild，期间一直黑屏。需要加 debounce（比如 orientation 稳定 500ms 再触发）。

4. **passthrough 与 wda_mjpeg 并存时的代码维护**  
   `IosMjpegStreamer` 既是 wda_mjpeg 的完整实现，也是 `IosMjpegPassthroughStreamer` 的父类。加 `on_reset` / `rebuild_on_rotate` 参数时必须保证 passthrough 子类显式传 `rebuild_on_rotate=False`（或 `on_reset=None` 无动作）避免干扰。

## 6. 为什么现在不做

1. `mjpeg_passthrough`（Sonic 同款方案）已经**天然绕开**此问题——每帧独立 JPEG，`<img>.naturalWidth/Height` 在 `load` 时自动更新，零链路重建即可适配任意方向
2. 当前默认后端就是 passthrough，清晰度更高、延迟更低、Mac CPU 更低、旋转自适应，**没有任何场景需要回 wda_mjpeg**
3. 这份方案书一旦实施，会引入一条需要维护的冷路径（测试覆盖、CI、升级 Appium WDA 时重新验证）。**过早落地 = 持续成本**
4. 真正需要回 wda_mjpeg 的场景（例如未来浏览器对 `<img>` Blob 连续重建有性能瓶颈、或某些 iOS 版本 JPEG 流不稳），到时候按这份文档实施也不迟——WDA 源码、Appium settings 语义到那时可能已变，现在存的代码也未必能直接用

## 7. 用户原始洞察（2026-04-20 对话）

> 我的 iOS 的 run 截图横竖正常，只是 web 端显示不对，并且每次遇见这个问题，我都是退出到首页，再进入就好了，这说明什么？重新进入手机页就修正了（甚至直接刷新页面也修正了），并且什么都没断，可以点击操作。

这个观察精准指出了修复方向：**wda_mjpeg 路径要解决旋转，必须同时重建"WDA 这端的 canvas 来源" + "浏览器这端的 MSE 消费者"**，单独动任一端都不够。本方案就是把这个手动操作固化成 agent 自动流程。
